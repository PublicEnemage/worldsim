## 3. Observable Application State

### 3.1 Primary observable state

With the Jordan/Egypt Hormuz fixture (or any completed JOR scenario) loaded in Zone 1, navigated to step 8, at viewport 1280×900:

The element `[data-testid="zone-1d-four-framework"]` contains four framework rows. Each row has a child element `[data-testid="framework-annotation-{framework}"]` (where `{framework}` is `financial`, `human_development`, `ecological`, `governance`). Each annotation element contains text matching the pattern `[T{N}` (square bracket, T, digit) — confirming the tier annotation is present. At this viewport, all four rows plus annotations are visible in the Zone 1D allocated height without scroll.

### 3.2 Secondary observable states

**Secondary state A — Assumption surface strip**

With any completed scenario loaded (scenario has been advanced beyond step 0), with fiscal multiplier set to 1.30 (non-default): the element `[data-testid="assumption-surface"]` exists in the DOM, is not `display:none`, renders as a single line at max 24px height, and contains the text `Fiscal ×1.30`. The element is positioned between `[data-testid="scenario-identity-header"]` (Zone 0) and `[data-testid="instrument-cluster"]` (Zone 1). No user interaction is required to see this element.

**Secondary state B — Political Feasibility row in Zone 1D**

With a scenario where the political economy module is enabled (PE enabled in `modules_config.political_economy.enabled`), after scenario advance: `[data-testid="zone-1d-political-feasibility"]` is visible within `[data-testid="zone-1d-four-framework"]`. The element contains the text "Political Feasibility" and a percentage value followed by a tier label (e.g., "59% [T3 · political economy module]"). When PE is not enabled, `[data-testid="zone-1d-political-feasibility"]` is absent from the DOM entirely (not hidden, absent).

**Secondary state C — Zone 1B compact rows full indicator names**

With the Greece M8 Demo fixture loaded at step 6 (or any step with two TERMINAL alerts), at viewport 1280×800: `[data-testid="zone-1b-compact"]` contains at least two elements with `data-testid="compact-alert-row"` and severity "TERMINAL" visible in the viewport without scroll. Each compact row shows a full indicator name using the 24-character abbreviation set — no mid-word truncation (no trailing "..." in the middle of a word). The full indicator name fits within the compact row's column width without overflow.

**Secondary state D — Zone 1C PMM annotation**

With any completed scenario loaded: `[data-testid="zone-1c-pmm"]` contains a child element `[data-testid="pmm-annotation"]` with the text `[T3 composite · pre-cal]` verbatim (until the G6 Chief Methodologist interpretation anchor is filed). This text is visible without interaction.

**Secondary state E — HCL strip tier meaning expansion**

With any completed scenario loaded: `[data-testid="cohort-indicators-panel"]` contains at least one indicator row where the tier display reads `[T{N} · {tier_meaning}]` (e.g., `[T4 · model estimate]`) with the unit in parentheses (e.g., `(ratio)`). The raw tier code "T4" does not appear without the tier meaning expansion.

### 3.3 Silent failure detection

**Component 1 — Zone 1D annotation null tier:**
If the trajectory endpoint returns `confidence_tier: null` for a framework row, the annotation renders as `[—]` (em-dash in brackets) rather than an empty string or nothing. An empty annotation slot would be indistinguishable from a design choice to not annotate; `[—]` explicitly signals unavailability.
Observable distinguisher: `[data-testid="framework-annotation-{framework}"]` element contains the text `[—]` and does not contain `T` followed by a digit.

**Component 2 — Assumption surface unavailable:**
If the scenario configuration cannot be parsed for assumption inputs (null `activeScenarioDetail`, `modules_config` absent, or no fiscal multiplier): `[data-testid="assumption-surface-unavailable"]` is present in the DOM in place of `[data-testid="assumption-surface"]`. The element is not display:none. An empty assumption surface strip (visible strip with no text content) is a silent failure — distinguishable from the unavailable state only if the `assumption-surface-unavailable` testid is present.

**Component 3 — PE enabled + null programme_survival_probability:**
If the PE module is enabled but the survival probability computation returns null (API returns null value for `programme_survival_probability`): `[data-testid="zone-1d-political-feasibility"]` is present (row is not suppressed when PE is enabled) and contains the text "Political Feasibility" followed by "— [computation error]". This distinguishes a computation failure from PE being disabled (which suppresses the row entirely).

## 4. Acceptance Criteria

**AC-1 (Zone 1D L0 annotations — present at zero interaction):**
In any completed scenario loaded in Zone 1, at step 1 or later, at viewport 1280×900, without any user interaction: each of the four elements `[data-testid="framework-annotation-financial"]`, `[data-testid="framework-annotation-human_development"]`, `[data-testid="framework-annotation-ecological"]`, `[data-testid="framework-annotation-governance"]` is present in the DOM, is not `display:none`, and contains a text string matching `[T` followed by a digit. All four annotation elements are within the visible viewport height without scroll.

**AC-2 (Zone 1D L0 annotations — tier, source, and pre-cal content):**
In the Hormuz scenario (JOR entity, step 8), at viewport 1440×900: `[data-testid="framework-annotation-financial"]` contains the text "T2" (the confidence tier integer for JOR Financial) AND contains a non-empty source description string that includes a source institution name (e.g., "IMF" or "CBJ") AND contains the text "pre-cal". The exact content depends on the data-quality endpoint response for JOR, but these three substrings must all be present. Assert string presence for each: `expect(annotationText).toContain("T2")`, `expect(annotationText).toContain("pre-cal")`, and the institution name assertion uses a direct string match, not a character-class regex (NM-045 lesson).

**AC-3 (Zone 1D L0 annotation — ecological breach type):**
In the Hormuz scenario (JOR entity, step 8), or any scenario with an ecological TERMINAL alert: `[data-testid="framework-annotation-ecological"]` contains text that includes either "ceiling" or "floor" verbatim (per ADR-015 EL Decision 2 — breach type must be specified in human-readable text, not only a directional arrow). The annotation must NOT contain only `[↑ = approaching breach]` without specifying ceiling or floor.

**AC-4 (Zone 1D L0 annotation — null tier → [—] fallback):**
When `GET /api/v1/scenarios/{id}/trajectory` is intercepted via `page.route()` and the response is modified so that `frameworks[0].confidence_tier` is null for the `financial` framework at step 1: `[data-testid="framework-annotation-financial"]` is present and its text content contains `[—]`. It does not contain `[T` followed by a digit. The remaining three framework annotations render normally (not affected by the financial null).

**AC-5 (Assumption surface — present and contains fiscal multiplier):**
With a scenario where fiscal multiplier is set to 1.30 (non-default), after the scenario is loaded in Zone 1: `[data-testid="assumption-surface"]` exists in the DOM, is not `display:none`, and its text content contains `Fiscal ×1.30` — the multiplication sign (×, U+00D7) and the exact decimal value `1.30`. The element is visible in the viewport without any user interaction. `[data-testid="assumption-surface-unavailable"]` must NOT be present in the DOM when the assumption surface is rendering normally.

**AC-6 (Assumption surface — single-line height constraint):**
With any scenario loaded and the assumption surface visible: the rendered height of `[data-testid="assumption-surface"]` is ≤ 24px at viewport widths of 1280, 1440, and 1920. If content overflows, ellipsis truncation is applied — no wrapping. Assert via `element.getBoundingClientRect().height <= 24`.

**AC-7 (Assumption surface — PE enabled text present when PE is active):**
With a PE-enabled scenario loaded (political economy module enabled in configuration): `[data-testid="assumption-surface"]` text content contains the substring "Political economy: enabled". With a scenario where PE is not enabled, the assumption surface does NOT contain "Political economy" text. Assert string presence/absence by direct string match.

**AC-8 (Assumption surface — unavailable fallback):**
When `GET /api/v1/scenarios/{id}` (scenario detail) is intercepted and returns a response with `configuration: {}` (empty configuration): `[data-testid="assumption-surface-unavailable"]` is present in the DOM. `[data-testid="assumption-surface"]` must NOT be present.

**AC-9 (Political Feasibility row — present when PE enabled):**
With a PE-enabled scenario loaded (political economy module enabled, scenario advanced at least one step): `[data-testid="zone-1d-political-feasibility"]` is visible within `[data-testid="zone-1d-four-framework"]`. The element's text content contains "Political Feasibility" verbatim, contains a percentage value (digits followed by "%"), and contains "T3" (or the actual tier for the PE module). Assert each by direct string match.

**AC-10 (Political Feasibility row — absent when PE disabled):**
With a scenario where the political economy module is NOT enabled (default scenario, no `modules_config.political_economy.enabled`): `[data-testid="zone-1d-political-feasibility"]` is absent from the DOM entirely. `await page.locator('[data-testid="zone-1d-political-feasibility"]').count()` must equal 0.

**AC-11 (Political Feasibility row — computation error fallback):**
With a PE-enabled scenario loaded, when the `programme_survival_probability` API response is intercepted and the value field is set to null (mocked via `page.route()`): `[data-testid="zone-1d-political-feasibility"]` is present and its text content contains both "Political Feasibility" AND "[computation error]". The row is NOT absent (PE is enabled — absence would imply PE is disabled). Assert direct string presence.

**AC-12 (Zone 1B compact rows — full indicator names visible at 1280×800):**
With the Greece M8 Demo fixture loaded at a step with two TERMINAL alerts, at viewport 1280×800: `[data-testid="zone-1b-compact"]` contains at least two elements with `data-testid="compact-alert-row"` visible in the viewport without scroll. Each compact row's indicator name text does NOT contain an ellipsis "…" in the middle of a word (mid-word truncation is prohibited). The indicator name text for `reserve_coverage_months` alert must contain the word "Reserve" and the word "Coverage" as separate readable words. Assert using the 24-char abbreviation defined in §4b.

**AC-13 (Zone 1C PMM annotation — pre-cal placeholder present):**
With any completed scenario loaded, with Zone 1C PMM visible: `[data-testid="pmm-annotation"]` exists and its text content contains `[T3 composite · pre-cal]` verbatim (exact string, including the middle dot ·). No variation in spacing or punctuation is acceptable — the QA Lead must assert exact string equality for this text.

**AC-14 (HCL strip — unit and tier meaning expansion):**
With any completed scenario loaded that has at least one non-null HCL indicator: at least one element with `data-testid` matching `cohort-tier-{key}` contains a string matching `[T{digit} · ` followed by one of: "real observed data", "official statistics", "synthetic", "model estimate", "synthetic extrapolation". The raw tier code alone (e.g., just "T4" without the meaning expansion) must NOT appear as the complete tier label — it must be followed by the meaning text.

### Component 1 — Zone 1D ecological annotation (EL Decision 2)

The ecological annotation must specify breach type (ceiling vs. floor) in human-readable text per EL Decision 2. The breach type is NOT currently in any API response — it is a static property of the scenario type:
- Greece-type scenarios (CO₂ boundary overshoot): `[↑ approaching planetary ceiling — climate]`
- Hormuz-type scenarios (resource depletion): `[↑ approaching resource floor — freshwater]`

**Implementation approach:** Static mapping from scenario entity + ecological indicator key to breach type label. No new API field required in M14 (schema-level Option B distinction is M15). Mapping maintained in frontend `ECOLOGICAL_BREACH_TYPES` constant.

**Data Architect must confirm:** The frontier condition for each entity's ecological breach type is:
- GRC, EGY scenarios: CO₂ boundary proximity → `ceiling` type
- JOR, ZMB scenarios: resource depletion proximity → `floor` type (confirm this mapping before implementation)

### Component 3 — Programme survival probability

| Field needed | Endpoint | Field path | Status |
|---|---|---|---|
| `programme_survival_probability` value | **PENDING DATA ARCHITECT DECISION** | NOT in trajectory endpoint. Available via `GET /scenarios/{id}/measurement-output?entity_id={id}&step={N}` → `outputs.political_economy.indicators.programme_survival_probability.value` | ⚠️ BACKEND DECISION REQUIRED |
| `confidence_tier` for PE row | **PENDING DATA ARCHITECT DECISION** | `outputs.political_economy.indicators.programme_survival_probability.confidence_tier` via measurement-output | ⚠️ BACKEND DECISION REQUIRED |
| `political_economy_enabled` flag | `GET /scenarios/{id}` | `configuration.modules_config.political_economy.enabled` (same as Component 2) | ⚠️ PATH UNKNOWN — same gap as Component 2 |

**Data Architect must decide before Component 3 implementation begins:**
The sprint entry asserts "this value is already present in the trajectory response from the political economy module (G6, M13). No backend change required." Based on intent authorship analysis:

- The trajectory endpoint (`/scenarios/{id}/trajectory`) returns `political_economy` as a framework entry with `composite_score` (the 3-factor composite from ADR-013) — NOT `programme_survival_probability` individually.
- `programme_survival_probability` is a named indicator in the political economy module state, accessible via `GET /scenarios/{id}/measurement-output?entity_id={id}&step={N}` → `outputs.political_economy.indicators.programme_survival_probability`.
- These are two distinct values: `composite_score ≈ 0.5650` vs `programme_survival_probability ≈ 0.5950` (confirmed by G6 BPO validate 2026-06-12).

**Three options:**

**Option A (preferred for no-backend-change):** Fetch `programme_survival_probability` from `measurement-output` endpoint at the current step. Zone 1D subscribes to step changes and fetches `measurement-output?entity_id={entity}&step={current_step}` for `political_economy` indicators. This is the same pattern already used for the HCL panel (`hdState` in `ScenarioInstrumentCluster.tsx` lines 227–232). Data Architect must confirm that `political_economy` framework indicators are returned by measurement-output at each step and confirm the field path.

**Option B:** Add `programme_survival_probability` as a named field to `TrajectoryFrameworkPoint` for the political_economy framework in the trajectory response. Requires backend change — Data Architect specifies new field in `api_contracts.yml`.

**Option C:** Use `political_economy` framework `composite_score` from trajectory as the Zone 1D Political Feasibility value. Simpler, no backend change, but displays the composite (0.5650) not the survival probability (0.5950). ADR-015 specifically requires `programme_survival_probability`, not the composite.

**Implementation may NOT begin for Component 3 until the Data Architect records a decision here and updates `api_contracts.yml` if Option A or B is chosen.**

## 5. Kryptonite Constraint Check

**Does this implementation's primary observable state require specialist mediation for Persona 2 to act on it in the Reactive entry state (90-second ceiling)?**

`[x]` No — the observable states are directly interpretable by Persona 2 without an analyst translating them.

Justification per component:

- **Component 1 Zone 1D annotations:** `[T2 · 4 indicators · IMF/CBJ · pre-cal]` — "T2" means "official statistics, citable directly" per ADR-015 §UX-5. The source institution name (IMF, CBJ) is recognizable to any finance ministry economist as a named data provider. "pre-cal" means "pre-calibration — we're completing the calibration documentation." No specialist translation required. The ministry team with three economists can respond: "This is Tier 2 observed data from the Central Bank of Jordan."

- **Component 1 PMM annotation:** `[T3 composite · pre-cal]` — "T3" means "early estimate, confirm before citing." "composite" tells the user this is a synthesized index. "pre-cal" discloses the calibration status. A ministry economist can say: "The PMM is a Tier 3 composite pre-calibration estimate — we're in the process of validating it." No specialist mediation.

- **Component 1 HCL tier expansion:** `-0.25 (capability index) [T4 · model estimate]` — the unit is in parentheses, the tier meaning is spelled out. A finance ministry economist does not need a specialist to understand "model estimate" vs. "official statistics."

- **Component 2 assumption surface:** `Fiscal ×1.30 · Political economy: enabled · Data: 2024-Q1 vintage` — plain-language display of the model inputs. No jargon, no specialist translation. The ministry team can read this directly.

- **Component 3 political feasibility:** `Political Feasibility  59% [T3 · political economy module]` — a percentage is unambiguous. "T3" with the source "political economy module" tells the user this is a module-computed estimate. 59% is directly usable at the table: "The model shows a 59% programme survival probability under these conditions."

Kryptonite test: every G5 observable state is readable by a three-economist finance ministry team and actionable at the negotiating table without specialist mediation from a fourth person who understands the model internals.
