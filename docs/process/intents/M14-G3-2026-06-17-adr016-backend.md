## 2. Persona Trace Elements Targeted

> *Derived from ADR-016 §Persona and UX Traceability.*

**P-1 — Persona served:**
Primary: Persona 2 — Finance Ministry Negotiator (Zambian ministry analyst archetype).
The analyst building a scenario before a negotiation session and responding to input
challenges at the table.

Secondary: Persona 3 — Policy Analyst / Country Economist (Kofi Otieno archetype).
The analyst reviewing a colleague's scenario post-creation, requiring source citations
for the model's starting conditions.

G3 is backend infrastructure — the endpoints and source registry data that G4 (frontend)
reads from. The persona trace flows through G4; G3's acceptance criteria are confirmed
at the API boundary, not the UI boundary. G4 cannot be implemented or verified without G3.

**P-2 — Entry state:**
Reactive (primary): Persona 2 in the negotiation room receives an input challenge
("your Jordan current account assumption is wrong"). She opens the Grounding strip
(one interaction). Within 3 seconds she reads a source-cited starting condition
derived from the `/initial-state` endpoint. Without G3, this screen is empty — the
endpoint does not exist and G4 cannot render the grounding strip.

Preparatory (secondary): Persona 2 building a scenario selects ZMB from the entity
selector and immediately sees a data quality preview derived from the `/data-quality`
endpoint. Without G3, the preview cannot load.

**P-3 — Journey reference:**
Journey A Step 2 [Near-Term-Gap] — data quality disclosure at scenario creation.
G3's `/data-quality` endpoint is the backend half of Journey A Step 2; G4 frontend
renders it.

Journey B Step 0 (new, introduced by ADR-016) — input challenge response in Reactive
state. G3's `/initial-state` endpoint is the backend half of Journey B Step 0.

**P-4 — Time/interaction ceiling:**
The `/data-quality` endpoint must respond within 2 seconds so the frontend can display
the preview within the Preparatory ceiling for entity + year selection. The
`/initial-state` endpoint must respond within 3 seconds so the Grounding strip meets
ADR-016 §UX-3 criterion 3 (visible within 3 seconds of opening).

These are ADR-016 P-4 ceilings for the frontend surface. G3's obligation: both
endpoints must complete their database queries within 500ms under single-user load
(leaving 1.5s–2.5s for network + frontend rendering).

**P-6 — Negotiating leverage delivered:**
After G3 is implemented, Persona 2 can make the following specific argument:

> "The model uses Jordan's current account deficit as reported in the IMF 2023
> Article IV Consultation — 6.8% of GDP as of year-end 2022. That figure is Tier 2
> confidence — observed data, not synthetic. If the creditor side uses a different
> figure, they need to produce their source. Here is ours."

Previously: Persona 2 could not identify what current account figure the model used,
from what source, or at what vintage, from any screen in the application. G3 is the
data layer that makes this argument possible; G4 is the display layer that puts it on
screen. Neither is sufficient without the other.

**P-7 — North star capability delivered:**
The `/initial-state` endpoint returns the source-cited starting conditions for the
Jordan (and Zambia) scenario, enabling the Grounding strip in G4 to close the
90-second input challenge response gap. The `/data-quality` endpoint enables the
pre-creation preview that signals to the Zambian analyst whether her 2024 data is
IMF-observed or synthetic inference before she commits to the scenario.

This intent document delivers P-7 at the API layer. Full P-7 (screen-visible,
within-90-seconds) requires G4 frontend to land on top of G3 backend.

## 3. Observable Application State

> *G3 is backend infrastructure. All observable states below are confirmable using
> `curl` or `httpx` against the running API at localhost:8000 — no browser, no source
> code reading.*

### 3.1 Primary observable state — `/data-quality` endpoint

`GET /api/v1/entities/JOR/data-quality?year=2024` returns HTTP 200 with a JSON body
matching the ADR-016 §Component 1 API contract:

```json
{
  "entity_id": "JOR",
  "year": 2024,
  "frameworks": [
    {
      "framework": "financial",
      "confidence_tier": 2,
      "source_institution": "IMF BOP",
      "data_vintage": "2024-Q1",
      "is_synthetic": false
    }
  ]
}
```

The `frameworks` array must contain **at least one entry** with a non-null
`confidence_tier` (integer 1–5) and a non-null `framework` string. The response must
not be a 404 or 500 for any entity in the set {GRC, JOR, EGY, ZMB}.

`GET /api/v1/entities/ZMB/data-quality?year=2024` returns HTTP 200 with at least one
framework entry where `is_synthetic: true` and `synthetic_basis` is a non-null,
non-empty string (e.g., "SADC comparable economies 2022–2023").

### 3.2 Secondary observable states

**Secondary state A — `/initial-state` endpoint**

`GET /api/v1/scenarios/{jor_scenario_id}/initial-state` where `jor_scenario_id` is a
completed or running JOR scenario, returns HTTP 200 with a JSON body matching the
ADR-016 §Component 2 API contract:

```json
{
  "scenario_id": "...",
  "entity_id": "JOR",
  "step_0_year": 2023,
  "frameworks": {
    "financial": {
      "indicators": [
        {
          "name": "reserve_coverage_months",
          "display_name": "Reserve coverage",
          "value": 3.2,
          "unit": "months",
          "source": "IMF BOP",
          "vintage": "2024-Q1",
          "confidence_tier": 2,
          "is_synthetic": false
        }
      ]
    }
  }
}
```

The `frameworks` object must contain at least one framework key with at least one
indicator. Each indicator object must have non-null `display_name`, `value`,
`confidence_tier`. `source` and `vintage` may be null only if `is_synthetic: true`
and `synthetic_basis` is non-null.

**Secondary state B — `api_contracts.yml` updated**

`docs/schema/api_contracts.yml` contains entries for both endpoints. The Data Architect
Agent confirms the schema matches ADR-016 §Component 1 and §Component 2 API contract
specifications verbatim. This is verifiable by reading the file — the two endpoint
entries must be present in the endpoints array before the G3 PR is marked ready for
review.

**Secondary state C — Source registry population**

For each entity in {GRC, JOR, EGY, ZMB}, the `/data-quality` endpoint returns at
least T3-quality data for the Financial and Human Development frameworks. That is:
`confidence_tier ≤ 3` for at least one Financial framework entry and at least one
Human Development framework entry per entity.

For ZMB, T4 synthetic data is acceptable if no T1–T3 source is available — but the
`is_synthetic` flag must be `true` and `synthetic_basis` must be non-null.

### 3.3 Silent failure detection

ADR-016 §Silent Failure Mode specifies two backend-relevant failure modes:

**SF-1 — Empty frameworks array (source registry incomplete):**
If no source registry entry covers a given entity/year combination, the endpoint must
return HTTP 200 with `"frameworks": []` — not a 404, not a 500, and not a response
with framework entries where `source_institution: null` and `is_synthetic: false`
simultaneously (which would imply observed data with no named source — a data integrity
violation). Detection: `curl GET /api/v1/entities/XYZ/data-quality?year=2024` (entity
not in source registry) must return `{"entity_id":"XYZ","year":2024,"frameworks":[]}`.

**SF-2 — Empty frameworks in `/initial-state` (scenario predates G3):**
If the scenario's step 0 snapshot has no source-registry-linked indicator data (e.g.,
a pre-G3 scenario), the endpoint returns HTTP 200 with `"frameworks": {}` — not a
404. Detection: `curl GET /api/v1/scenarios/{legacy_id}/initial-state` where
`legacy_id` is a pre-G3 scenario must return 200 with empty frameworks, not 404.

## 4. Acceptance Criteria

> *Backend pytest coverage — each AC below is testable with `httpx` or `asyncpg`
> against the running application. No browser interaction required.*

**AC-1 (data-quality — JOR financial data present):**
`GET /api/v1/entities/JOR/data-quality?year=2024` returns HTTP 200. The response body
`frameworks` array contains at least one entry where `framework == "financial"` and
`confidence_tier` is an integer between 1 and 5 inclusive.

**AC-2 (data-quality — ZMB synthetic flag present):**
`GET /api/v1/entities/ZMB/data-quality?year=2024` returns HTTP 200. At least one
entry in `frameworks` has `is_synthetic == true` and `synthetic_basis` is a non-empty
string.

**AC-3 (data-quality — GRC financial at T2 or better):**
`GET /api/v1/entities/GRC/data-quality?year=2010` returns HTTP 200. The entry with
`framework == "financial"` has `confidence_tier <= 2` (T1 or T2 — GRC 2010 has
IMF/World Bank observed data).

**AC-4 (data-quality — all four entities return 200):**
`GET /api/v1/entities/{entity_id}/data-quality?year=2024` for each entity in
{GRC, JOR, EGY, ZMB} returns HTTP 200. None returns 404 or 500. (Year 2024 may
not have data for all frameworks — empty frameworks array is acceptable; 404/500
is not.)

**AC-5 (data-quality — unsupported entity returns empty frameworks, not 404):**
`GET /api/v1/entities/XXX/data-quality?year=2024` (entity code not in source
registry) returns HTTP 200 with `"frameworks": []`. The response must not be HTTP
404 or 500. The `entity_id` field in the response body is `"XXX"` and `year` is
`2024`. (SF-1 detection criterion.)

**AC-6 (initial-state — JOR scenario returns indicator with source citation):**
Given a JOR scenario that has been run (status `"completed"`), `GET
/api/v1/scenarios/{jor_scenario_id}/initial-state` returns HTTP 200. The response
body contains `entity_id == "JOR"`. The `frameworks` object contains at least one
key (e.g., `"financial"`) with at least one indicator entry. That indicator entry has
a non-null `display_name` and `confidence_tier` between 1 and 5.

**AC-7 (initial-state — reserve_coverage_months indicator cited):**
Given a JOR scenario (status `"completed"`), `GET
/api/v1/scenarios/{jor_scenario_id}/initial-state` returns an entry in
`frameworks["financial"]["indicators"]` where `name == "reserve_coverage_months"`.
That entry has a non-null `source` (e.g., `"IMF BOP"`) and a non-null `vintage`
(e.g., `"2024-Q1"`). This is the specific indicator that Persona 2 cites in the
north star scenario when responding to an input challenge.

**AC-8 (initial-state — legacy scenario returns empty frameworks, not 404):**
Given a scenario created before G3 (a pre-G3 scenario with no source-registry-linked
step 0 state), `GET /api/v1/scenarios/{legacy_id}/initial-state` returns HTTP 200
with `"frameworks": {}`. The response must not be HTTP 404. (SF-2 detection
criterion.)

**AC-9 (api_contracts.yml — both endpoints present):**
`docs/schema/api_contracts.yml` contains two new endpoint entries: one for
`GET /api/v1/entities/{entity_id}/data-quality` and one for
`GET /api/v1/scenarios/{scenario_id}/initial-state`. Each entry includes `path`,
`method`, `summary`, `path_params`, `query_params` (where applicable), `response`
with `status: 200` body schema, and `db_reads`. The schema must match ADR-016
§Component 1 and §Component 2 API contract specifications. (Verifiable by reading
`docs/schema/api_contracts.yml` after the PR merges.)

## 5. Kryptonite Constraint Check

**Does this implementation's primary observable state require specialist mediation for
Persona 2 to act on it in the Reactive entry state (90-second ceiling)?**

`[x]` No — the observable state is a JSON API response read by the G4 frontend, which
renders it as the Grounding strip label `"[Indicator]: [value] [unit] · [Source ·
Vintage · T{N}]"`. That format (specified in ADR-016 §Component 2) is interpretable
by Persona 2 without specialist mediation: the source name and vintage date are human-
readable, and T{N} is a single-character tier marker whose meaning is disclosed on
first use.

Kryptonite test: the ministry-side analyst with three economists can read
"Reserve coverage: 3.2 months · IMF BOP 2024-Q1 · T2" and cite the IMF 2024 BOP
source in response to an input challenge. The creditor-side workflow requires the same
API data interpreted the same way. No mediation asymmetry is introduced.

The G3 backend does not expose raw database field names or internal tier codes to the
user — it returns `display_name` (human-readable) and `confidence_tier` (integer
1–5). The frontend (G4) maps tier to the `T{N}` label. No specialist is required to
interpret a number between 1 and 5.

## 6. Out of Scope

**ADR-016 Component 3 (Fidelity panel contextualisation):** Deferred to M15 by EL
decision (ADR-016 §Decision 2). G3 does not implement analogous-case selection logic
or the Chief Methodologist similarity algorithm. The Fidelity panel receives a static
explanatory header in G4; that header requires no new backend endpoint.

**ADR-016 Component 4 (parameter persistence):** G4 scope. Component 4 reads from
`scenarios.configuration` (already stored); it does not require a new backend endpoint
from G3. G3 is not responsible for Component 4.

**Source registry population beyond the four M14 entities (GRC, JOR, EGY, ZMB):**
EL decision (ADR-016 §Decision 1). Entities beyond the four-entity M14 scope are out
of scope. The endpoint must handle requests for other entity codes gracefully (empty
frameworks, not 404), but source registry seeding is only required for GRC, JOR, EGY,
ZMB.

**Real-time data fetching or external API calls:** The `/data-quality` endpoint reads
from the `source_registry` table only — it does not make external calls to IMF, World
Bank, or other sources at request time. Source registry data is populated offline
(migration or seed script) before the endpoint is called.

**Per-cohort initial state disaggregation:** ADR-016 §Known Limitations. The
`/initial-state` endpoint returns nation-level indicator values; it does not
disaggregate by income cohort or demographic subgroup. That capability is M15+.

**CI_lower / CI_upper banding on initial state indicators:** ADR-007 pending. G3 does
not return confidence interval bands on indicator values; it returns `confidence_tier`
(the data quality tier) only.
