## 2. Persona Trace Elements Targeted

*Derived from the Zone 1A Phase 1 design thinking document and the governing north-star question
for the ADR-017 panel. ADR-017 does not implement the encoding — it specifies the encoding that
Phase 4 will implement. The persona trace below names the capability the accepted ADR-017 must
be specified to deliver.*

**P-1 — Persona served:**
Persona 2 — Finance Ministry Negotiator (Eleni/Aicha archetype). The binding constraint is the
Mode 3 question: can the finance ministry economist read Zone 1A's direction-of-effect signal
within 15 seconds of a control input being applied, for a scenario with N≤4 entities — without
specialist mediation? The ADR-017 encoding specification must satisfy this constraint at panel
acceptance time; Phase 4 implementation must satisfy it at the observable-state level.

**P-2 — Entry state:**
Reactive entry state (90-second ceiling, negotiating room context) for the Zone 1B/1D companion
instruments. For Zone 1A specifically: 15-second sub-ceiling for Mode 3 direction-of-effect read;
30-second sub-ceiling for Mode 1/2 threshold-crossing read. These are the time ceilings the
ADR-017 encoding design must be demonstrated (at panel review time) to satisfy.

**P-3 — Journey reference:**
- Journey B Step 3 [Near-Term-Gap]: Persona 2 defends a challenged output. "At which step did
  the threshold cross — and which framework?" (Mode 1). The ADR-017 encoding must answer this
  from Zone 1A without interaction for single-entity; from Zone 1A + entity selector for
  multi-entity (no navigation away from instrument cluster required).
- Journey A Step 2 [Preparatory]: Persona 2 checks trajectory shape before the session. Mode 2
  threshold-safe path question ("does any framework project a crossing?") must be answerable from
  Zone 1A at each step advance, 30-second ceiling.
- Mode 3 (no named Journey step yet — Demo 6 context): Persona 2 applies a control input in a
  live scenario and reads the direction-of-effect signal ("did this help or hurt?") within 15
  seconds.

**P-4 — Time/interaction ceiling:**
- Mode 1 (single entity): 30 seconds, zero additional interaction beyond loading the scenario
- Mode 2 (single entity): 30 seconds from each step advance
- Mode 3 (N≤4 entities): 15 seconds from control input application — this is the binding
  constraint for the ADR-017 encoding design
- Multi-entity (Mode 1/2, N>1): 30 seconds from loading the scenario, entity selector permitted
  as one interaction for per-framework follow-up (entity selector is in the persistent header,
  no navigation away from Zone 1 required)

**P-6 — Negotiating leverage delivered:**
ADR-017 accepted means: the architecture specifies an encoding where Persona 2, in the Mode 3
live scenario context, can answer "did the control input I just applied move our trajectory
away from or toward the nearest MDA threshold breach?" without asking a specialist to interpret
the chart. The accepted ADR-017 must specify this capability explicitly in its north star test
(Element P-7) — not only imply it through the encoding description.

**P-7 — North star capability delivered:**
The ADR-017 specification, when implemented in Phase 4, will enable the Zambia finance ministry
economist in a live Mode 3 control session to read the effect of a fiscal multiplier adjustment
on the composite trajectory — per entity, baseline vs. active — within 15 seconds, for N≤4
entities simultaneously, with no specialist present to translate the chart. The accepted ADR-017
names this scenario (or an equivalent named scenario) in its P-7 element and confirms the
15-second ceiling is satisfied at N=1 through N=4 under the specified encoding.

## 3. Observable Document State

> **Adaption for design sprint:** The "observable state" for G2 is the state of filed documents
> at their canonical paths, verifiable by reading those files — not by running the application.
> Each statement below can be confirmed by a QA reviewer using `find`, `grep`, and file reads,
> with no knowledge of the documents' intellectual contents. Whether the encoding is *correct*
> or *sufficient* is a Business PO Validate question (Step 5); whether the documents *exist and
> contain required structural elements* is the QA verifiable question (Step 4).

### 3.1 Primary observable state

`docs/architecture/reviews/ARCH-REVIEW-007-milestone15.md` exists at its canonical location
(`docs/architecture/reviews/`, matching the naming convention of ARCH-REVIEW-001 through
ARCH-REVIEW-006), AND `docs/adr/ADR-017-zone-1a-information-architecture.md` exists, AND
`docs/architecture/backlog.md` contains the string `ASSIGNED — ADR-017` in the ARCH-011 row
(replacing `PENDING_NUMBER`), AND ADR-017's UX Designer sign-off block contains all four NM-042
required fields with a declared `Session context` value.

### 3.2 Secondary observable states

**Secondary state A — ARCH-REVIEW-007 Phase 2 completeness:**
ARCH-REVIEW-007 contains explicit assessments of:
1. Question 1 — Zone 1A's primary question in each mode (Modes 1, 2, and 3 each addressed)
2. Question 2 — which encoding channels the proposed design uses (per mode)
3. Question 3 — N and M limits before legibility breaks (specific numeric limits stated per mode)
AND ARCH-REVIEW-007 records a binding decision on all four open Phase 2 questions identified in
`docs/ux/design-thinking/zone-1a-information-architecture.md §Phase 2 Readiness`: (1) Mode 3
single-entity encoding choice, (2) multi-entity COMPARE_VIEW, (3) composite aggregation rule,
(4) endpoint label collision handling.

**Secondary state B — ADR-017 structural completeness:**
ADR-017 contains: frontmatter `tier:` field (Tier 1); a persona trace section with elements P-1
through P-7; a UX implication statement; a silent failure mode section; a north star test (P-7);
a mission impact statement; and a Mermaid diagram (mandatory per `docs/CODING_STANDARDS.md` for
all ADRs). ADR-017's EL acceptance is on record: either `el-approved:` in frontmatter with a
date, or a recorded EL acceptance comment on issue #845.

### 3.3 Silent failure detection

**ARCH-REVIEW-007 nominal vs. substantive:** The silent failure mode for the Architecture Review
is a document that mentions all three Phase 2 readiness questions and all four open questions
but does not make binding decisions — deferring each to "Phase 3 panel decision." An
ARCH-REVIEW-007 that defers the four open Phase 2 questions without resolving them is not a
completed Architecture Review; it is a restated list of open questions. Detection: AC-5 checks
that each of the four open questions has a resolved decision statement (not merely a restatement
of the question). QA check: `grep -c "Decision:"` (or equivalent resolution marker) in
ARCH-REVIEW-007 must return ≥4.

**ADR-017 NM-042 incomplete sign-off:** The silent failure mode for the UX Designer sign-off
is a checkbox without all four named fields, or a `Governing documents reviewed` field that
uses generic references ("governing premises", "first principles") rather than named sections.
Detection: AC-9/AC-10 check for specific field names and at least two named document sections
in the reviewed field. A sign-off without a declared `Session context` must be treated as
non-compliant per `CLAUDE.md §ADR template — mandatory starting point`.

## 5. Kryptonite Constraint Check

**Does this implementation's primary observable state require specialist mediation for Persona 2
to act on it in the Reactive entry state (90-second ceiling)?**

`[x]` No — the observable state is interpretable by Persona 2 without an analyst translating it.

**Rationale:** G2's primary output is an accepted ADR-017 — a design specification document.
The kryptonite constraint applies to the encoding that ADR-017 specifies, not to the document
itself. The Phase 2 Architecture Review and Phase 3 ADR authorship must ensure that the
specified encoding satisfies the kryptonite constraint for Phase 4 implementation; this is
verified at the Business PO Validate step (Step 5) by confirming that the ADR's P-7 north
star test names a specific scenario where Persona 2 reads Zone 1A's direction-of-effect signal
within 15 seconds without specialist mediation.

**Kryptonite constraint check for ADR-017 specification (required at Step 5):** The ADR-017
panel — specifically the Business PO's P-7 element — must answer: "Can Eleni (Persona 2),
in the Mode 3 live Hormuz scenario with JOR + EGY entities loaded, read whether the fiscal
multiplier adjustment helped or hurt the JOR composite trajectory versus the baseline, within
15 seconds, without the Frontend Architect or Chief Methodologist explaining the chart to her?"
If the answer is no under the specified encoding, the encoding does not satisfy the kryptonite
constraint and the ADR is not ready for EL acceptance.

## 9. Step 5 Business PO Validate Record

**Verdict:** ACCEPT — 2026-06-22

**Business PO:** Business PO Agent
**Work type:** Documentation / Architecture (Tier 1 ADR)
**Validate criterion applied:** Documentation — non-author can navigate to the key finding from
the document's entry point in under five minutes; and analytics — the specification changes
what Persona 2 can argue at the negotiating table.

**Documentation navigation test:**
ADR-017 §Visual Mockups links to the encoding mockup HTML file. The 6-panel SVG overview is
immediately legible without scrolling. The §Decision table (Mode 3 N≤4 row) and §North Star
Test (Zambia Mode 3 scenario) are reachable within 2 minutes of opening ADR-017. Non-author
navigation criterion: **PASS** (well under 5 minutes).

**Kryptonite constraint check (required per §5):**
The Mode 3 composite encoding reduces Zone 1A visual load from 16 lines (4 frameworks × 2
entities × 2 branches) to 4 lines (2 per entity: baseline ghost + active solid). The divergence
fill between ghost and active solid is the self-interpreting directional signal — "ZMB solid
above ghost baseline → fiscal adjustment improved Zambia's trajectory." Zone 1D delta annotations
("Financial: +0.04 vs baseline") provide the per-framework follow-up without interaction or
specialist mediation. Kryptonite constraint: **PASS**.

**North star test (P-7):**
The Zambia ministry analyst in a live Mode 3 session with JOR + ZMB loaded can state: "The
proposed fiscal multiplier improves Zambia's aggregate trajectory without crossing any MDA
threshold — we can accept this term." This specific argument was structurally unavailable before
ADR-017 because the 16-line Zone 1A encoding required specialist interpretation within the
15-second ceiling. ADR-017's composite encoding removes that specialist-mediation requirement at
the specification level. North star: **PASS**.

**Customer Agent Layer 3 check:**
N/A at ADR authorship stage — ADR-017 specifies future behavior; no live output exists yet.
Layer 3 gate is triggered at Phase 4 implementation (separate sprint entry). See intent §7 and
sprint exit §3.

**Open rejections:** None.

**Implementation gate carried to Phase 4 sprint entry:**
ADR-017 §Silent Failure Mode identifies Zone 1D delta annotations as a required companion to
Zone 1A composite encoding. Phase 4 implementation of Zone 1A composite without Zone 1D delta
annotations is architecturally incomplete — it answers "did it help?" while removing the
"which framework?" follow-up that completes the 90-second Reactive session. The Phase 4 sprint
entry must include Zone 1D delta annotation implementation in the same sprint, or provide an
EL exception with an explicit forward trace to the sprint that will implement it. This is not
a G2 rejection — ADR-017 specifies the dependency correctly and flags it as a silent failure mode.
It is a Phase 4 entry condition.

**Business PO verdict:** ACCEPT. ADR-017 specifies a Zone 1A encoding that, when implemented per
Phase 4, will enable Persona 2 to read the Mode 3 directional signal without specialist mediation
in 15 seconds — closing the most operationally significant asymmetry gap in the primary instrument.
G2 sprint exit is confirmed complete.
