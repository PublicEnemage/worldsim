### IR-S7-004 — PMM = 0.00 across all four steps

- **Severity:** 🟠 Significant
- **Instrument:** Zone 1C — PMM Widget
- **Observation:** "0.00 ↑" is the displayed value in all five frames (steps 1 through 4).
  A PMM of 0.00 throughout the entire four-step Argentina arc — from programme entry through
  sovereign default through Kirchner recovery — is not analytically meaningful. The PMM
  represents the distance from the nearest MDA threshold. A PMM of exactly 0.00 means all
  thresholds are simultaneously at the floor — or that the computation is producing a fixed
  output regardless of step state. Either is a bug.
- **Expected behaviour per brief:**
  - Step 1 (baseline): PMM should be positive, reflecting distance from thresholds at
    programme entry
  - Step 2 (default): PMM should decrease (crisis compresses margin)
  - Step 3 (recovery begins): PMM should show improved margin vs step 2
  - Step 4 (growth consolidation): PMM should reflect expanded margin
- **Two possible root causes:**
  1. The PMM computation finds no applicable MDA thresholds for the Argentina fixture (all
     thresholds are cohort-scoped or entity-scoped for GRC, not ARG). `_compute_pmm_for_step()`
     returns 0.0 when no applicable thresholds exist.
  2. The trajectory response is returning `pmm: null` or `pmm: {value: "0.00", ...}` for
     all steps — the frontend renders null PMM as "0.00" rather than "—".
- **Action:** Inspect the trajectory API response for the Argentina scenario at step 3.
  Check `steps[2].pmm` — is it null or `{value: "0.00000", direction: "up"}`? If null
  and the frontend renders null PMM as "0.00", that is a frontend null-treatment bug (should
  render "—", per the null PMM contract). If the API returns `value: "0.00000"`, investigate
  whether any `'all'`-scoped MDA thresholds exist for the Argentina fixture. The governance
  MDA floor (`democratic_quality_score ≤ 0.70`) should be 'all'-scoped and should produce
  a non-zero margin at step 1 (score = 0.71, floor = 0.70 — margin is 0.01).
- **Narration impact:** The walkthrough Step 6 narration does not currently name a PMM value
  ("The four axes are in the instrument cluster"). If PMM = 0.00 cannot be resolved before
  the demo, the walkthrough should not draw attention to Zone 1C. If PMM is "—" (null), the
  presenter should say "PMM activates once at least one threshold is breached and we advance
  one step — it is null at step 0." The current "0.00 ↑" display is the worst of both: it
  implies a computed value that is analytically meaningless.

### IR-S7-005 — Choropleth visual dominance relative to Zone 1 instruments

- **Severity:** 🟡 Minor
- **Surface:** Zone 2 / Zone 1 viewport balance
- **Observation:** In all five frames, the choropleth map occupies approximately 60–65% of
  total viewport height. The Zone 1 instrument cluster (all four instruments combined)
  occupies the upper 35–40%. At a conference screen or projected display, the audience's
  eye will be drawn to the geographically prominent choropleth — the larger, more colourful,
  visually dominant element — rather than to the Zone 1 instruments where the analytical
  argument lives.
- **This is not a narration violation.** The updated walkthrough (PR #645) is disciplined —
  it does not direct the presenter to narrate the choropleth as the thesis visualization
  (UX-RULING-4). The narration correctly directs to Zone 1A at every step.
- **The issue is visual weight, not narration.** An audience watching a presenter point to
  a small panel in the upper-left while a full-world choropleth fills most of the screen
  may struggle to follow. The layout is architecturally correct (choropleth as navigable
  context, Zone 1 as always-visible instruments) but the visual emphasis balance may not
  serve Demo 3's argument at presentation resolution.
- **Action:** This is not a blocker for Demo 3. For the live session, ensure the presenter
  explicitly orients the audience to Zone 1 at the start ("the analytical instrument is
  the panel on the left — the map provides geographic context") — the walkthrough Step 1
  already includes this orientation. No implementation required before Demo 3. File as M11
  layout enhancement if the live session confirms the imbalance matters to observers.
