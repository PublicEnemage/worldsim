**DEMO-156: Frame C identical to Frame A — Act 1 Finding frame does not exist as a distinct capture**

Severity: CRITICAL
Confirmed by: All 9 agents (QA Lead CRITICAL; Frontend Architect CRITICAL; Customer Agent CRITICAL; UX Designer CRITICAL; Business PO CRITICAL; Development Economist CRITICAL; Data Architect CRITICAL; UX Design Thinking Agent CRITICAL; Chief Methodologist: same state at both captures — no distinct finding visible)
Frames: A, C
What is shown: `frame-c-act1-finding.png` and `frame-a-the-instrument.png` share identical MD5 (`ceaff34a7267ca19aa144d408e118614`) and byte count (193,781 bytes). Both show the same Zone 1A chart, same Zone 1B COHORT IMPACT rows (agricultural workers HIST, formal sector HIST), same Zone 1C PMM, same Zone 1D values, same control plane state.
What should be shown: The walkthrough Screenshot Reference specifies Frame C shows "Zone 1B CohortImpactSection — cohort threshold state / Bottom quintile informal workers poverty headcount at step 8 vs. 0.40 recovery floor. Finding: threshold crossed or avoided under tested multiplier." The Act 1 narration Section 3 (lines 309–349 of walkthrough) reads Zone 1B directly: "Bottom quintile informal workers. Poverty headcount. Recovery floor: 0.40." No distinct rendering of a focal informal workers row is present.
Root cause: Cluster C delivered `CohortImpactSection` monitored-row rendering capability (`FocalCohortConfig`, `focalRows`, CLEAR/MONITORED state logic). However, `monitoredFocalCohorts` is passed from `activeScenarioDetail?.configuration?.monitored_focal_cohorts` (`ScenarioInstrumentCluster.tsx:1040`). The backend schema defaults this field to `[]` (`backend/app/schemas.py:399`: `monitored_focal_cohorts: list[dict[str, Any]] = []`). The SEN demo scenario created by `createSenScenario` in the spec does not include `monitored_focal_cohorts` in its configuration payload. Therefore the component receives `[]`, `focalRows` is empty, and no monitored-row renders. Frame C and Frame A are captured at the same UI state with no component-level difference between them.
Fix: Add `monitored_focal_cohorts` to the SEN scenario configuration in the spec's `createSenScenario` call. The focal cohort entry should specify: `indicator: "bottom_quintile_informal_workers_poverty_headcount"`, `label: "Bottom Quintile Informal Workers — Poverty Headcount"`, `recovery_floor: 0.40`, `scenario_id: senId`. Verify the backend `ScenarioCreate` schema accepts and persists this field. If the backend doesn't yet support configuration-level focal cohort specification, add a DEV seam analogous to `__worldsim_selectedScenarioId` that injects the focal cohort config at runtime for the demo. After fix, Frame C capture must show the informal workers row with CLEAR or BREACHED state before recapture.

**DEMO-157: psp-driver-row absent from Frame A (step 8) — primary "THESIS" screenshot omits G2 deliverable**

Severity: CRITICAL
Confirmed by: 8 of 9 agents (Chief Methodologist rates HIGH — the omission is a walkthrough gap, not a methodology issue)
Frames: A
What is shown: Frame A Zone 1D Political Risk shows "Programme survival: WARNING (47%) — STABLE / Comparable: Ghana 2014 ECF — modified Step 5 (PSP 52%). Conditionality relaxed." No "Driver: fiscal sustainability" row appears between the severity badge and the comparable text.
What should be shown: Frame A is the walkthrough's primary "THESIS" screenshot (Screenshot Reference row 1). The walkthrough Presenter Briefing §What Is New in Milestone 18 item 2 states: "Zone 1D now shows the dominant causal driver category beneath the PSP severity badge at every step." Frame B (step 3) correctly shows "Driver: fiscal sustainability" in the same Zone 1D position. Frame A (step 8, the primary screenshot) does not.
Root cause: The trajectory route mock (`demo-narrated.spec.ts` line 246–247, 339–340) includes `psp_dominant_driver: "fiscal_sustainability"` at all steps `>= BRANCH_FROM_STEP`. This mock is served for the GET /trajectory call at app load time (step 3 state). Frame B is captured at step 3 from this mocked data — driver visible. Frame A is captured at step 8, after 5 real `POST /next-step` API calls to the backend (steps 4–8). The real backend's synthetic SEN data does not populate `psp_dominant_driver` in per-step responses because the SEN scenario's `_attribute_dominant_driver()` computation returns `None` for the synthetic fixture. The AC-D1 test validates the mock STRUCTURE (step 3 value present) but does not validate rendered output at step 8. The AC-D1 test passes while the rendering fails at step 8.
Fix: Two options. Option 1 (backend): Ensure the real backend's `next-step` response for the SEN demo scenario includes `psp_dominant_driver` by populating it in the PSP computation for synthetic Senegal data (`_attribute_dominant_driver()` must return `"fiscal_sustainability"` for the demo scenario's configuration). Option 2 (E2E): Extend the route mock to intercept `POST /next-step` calls and inject `psp_dominant_driver` in the political economy framework for steps 4–8 (same value as step 3). Option 1 is preferred (fixes the real backend, not just the E2E layer). Either way, Frame A must be recaptured with the driver row visible before the step 7 gate.

**DEMO-158: Choropleth centered on northern hemisphere in Frames D/E — ZMB (Zambia) not in viewport**

Severity: HIGH
Confirmed by: 7 of 9 agents (Chief Methodologist abstains — map centering is not a methodology issue; Development Economist MEDIUM — map is orientation, not analysis)
Frames: D, E
What is shown: The choropleth bottom panel in Frames D and E shows the northern hemisphere (Arctic Circle labels visible, Greenland/Canada/Russia in frame). Zambia (southern Africa) is not visible anywhere in the choropleth.
What should be shown: The choropleth should center on or include ZMB when presenting Act 2 (Zambia debt restructuring scenario). A Zambian Finance Minister stakeholder audience would notice the map not highlighting their country. AC-E4 in `demo-narrated.spec.ts` explicitly warns: "AC-E4: `__worldsim_getMapCenter` unavailable — ZMB centering is a visual-review check only." This warning fired during run 9.
Root cause: Cluster E fix added `page.evaluate(() => { const fn = (window).__worldsim_centerOnEntity; if (fn) fn('ZMB'); })` before Frame D capture. The DEV seam `__worldsim_centerOnEntity` does not exist in `App.tsx` — the function is `undefined` at runtime. The `if (fn)` guard silently skips the centering. No map pan occurs. AC-E4's `visual-review check only` designation correctly anticipated this but did not block recapture.
Fix: Implement the `__worldsim_centerOnEntity(entityId)` DEV seam in `App.tsx` (analogous to `__worldsim_selectedScenarioId`). The seam should invoke the choropleth component's `setCenter`/`flyTo` API for the given entity ID. Alternatively, invoke the pan via the Mapbox/Leaflet instance directly. After implementation, verify ZMB is centered in the choropleth before Frame D/E recapture. This is HIGH (not CRITICAL) because the distributional comparison argument does not depend on choropleth centering — but it is a credibility issue in a Zambian audience context.
